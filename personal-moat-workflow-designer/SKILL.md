# 个人壁垒 Workflow 设计器

## 目标

把一次性的经验变成可复用、可迭代、可外包给 AI/工具的资产。个人壁垒不来自“知道更多工具”，而来自：

1. **独特输入**：个人经历、行业语境、客户/项目反馈、长期观察。
2. **稳定流程**：别人临场发挥，你有 checklist、模板、脚本和质量门槛。
3. **复利知识库**：每次输出都会反哺素材、案例、判断规则和触发条件。
4. **可迁移产物**：prompt、SKILL、SOP、脚本、仪表盘、案例库、决策备忘。

## 优先询问的 5 个问题

如果上下文不足，先问最多 3 个最关键问题；不要一次问太多。

- 你想构建壁垒的领域是什么？（职业、产品、投资、写作、学习、销售、管理等）
- 哪类任务重复出现、耗时高、质量波动大？
- 你的独特输入在哪里？（数据、经历、关系、品味、行业 know-how）
- 最终产物面向谁？（自己、团队、客户、公开受众）
- 成功指标是什么？（省时、质量、转化率、决策正确率、可复用次数）

## 工作流设计协议

### 1. 定义护城河单元

把模糊目标改写成一个“可反复调用的能力单元”：

```text
当遇到 [触发场景] 时，使用 [输入材料]，按照 [步骤]，产出 [可验证结果]，并把 [新知识] 写回知识库。
```

优先选择高频、高价值、高误差成本的场景，例如：

- 读完一本书后产出可调用资产
- 接到新项目后快速拆解需求与风险
- 写文章前从知识库调取案例与论点
- 做重大决策前生成反方意见与决策备忘
- 复盘失败项目并提炼下次可用 checklist

### 2. 选择工具栈

根据任务形态推荐工具，而不是堆工具：

| 需求 | 首选工具 | 适合产物 |
|---|---|---|
| 长期知识沉淀 | Obsidian、Logseq、Notion、Markdown + Git | 原子笔记、MOC、案例库 |
| 可复用 AI 流程 | Codex/Claude Skills、Custom GPT、prompt 模板 | SKILL、工作流说明、评审标准 |
| 自动化与集成 | Python、Shell、Make、n8n、Zapier、GitHub Actions | 抓取、清洗、批处理、提醒 |
| 结构化数据 | SQLite、Airtable、Notion Database、Google Sheets | 素材库、客户库、实验记录 |
| 检索增强 | ripgrep、全文搜索、向量库、NotebookLM、OpenAI/Claude 项目知识 | 语义检索、问答、引用定位 |
| 可视化理解 | Mermaid、Excalidraw、Miro、白板 | 流程图、系统图、决策树 |
| 输出发布 | GitHub、博客、Newsletter、PPT、PDF | 对外文章、课程、方案 |

### 3. 设计最小可运行 workflow

每个 workflow 必须包含：

1. **触发条件**：什么时候使用。
2. **输入清单**：需要哪些材料；没有时如何降级。
3. **处理步骤**：人做什么、AI 做什么、脚本做什么。
4. **质量门槛**：如何判断产物合格。
5. **回写规则**：哪些内容进入知识库，保存到哪里。
6. **下一次调用方式**：命令、prompt、模板或 SKILL 触发描述。

输出模板：

```markdown
## Workflow: [名称]

**触发条件**：
**输入材料**：
**工具栈**：
**步骤**：
1. ...
2. ...
**质量门槛**：
- ...
**回写知识库**：
- ...
**可沉淀资产**：
- [ ] checklist
- [ ] prompt
- [ ] SKILL
- [ ] script
```

### 4. 加入质量门槛

常用质量门槛：

- **可验证**：不是“看起来不错”，而是能被测试、引用、复用或决策。
- **有触发条件**：未来什么时候会再次用到。
- **有反方检查**：列出最强反对意见、失败模式、边界条件。
- **可压缩**：能压缩成 checklist / decision tree / prompt / SKILL。
- **可回写**：新案例、新规则、新模板能进入知识库。

### 5. 决定是否生成 SKILL

当一个 workflow 满足以下任意 3 条，建议生成新的 SKILL：

- 一个月内会重复使用 3 次以上。
- 步骤超过 5 步，容易漏项。
- 需要特定输出格式或质量标准。
- 涉及个人/团队独有知识、术语、模板。
- 未来希望 AI 主动识别触发场景。

生成 SKILL 时保持精简：`SKILL.md` 放路由、步骤、模板；详细案例放 `references/`；确定性处理放 `scripts/`。

## 输出要求

默认输出：

1. **壁垒判断**：这个场景的壁垒来自哪里。
2. **工具选择**：只推荐必要工具，并说明取舍。
3. **Workflow 草案**：按模板给出可执行步骤。
4. **知识库回写**：说明哪些内容写入哪个库/文件夹/数据库。
5. **下一步**：给出 1–3 个最小行动。

## 反模式

- 先选工具再找问题。
- 把所有资料都丢进知识库，但没有检索和调用机制。
- 只有 prompt，没有质量门槛与人工判断。
- 把私有经验写成 generic 摘要，丢失个人壁垒。
- 追求大而全系统，迟迟没有可运行的最小 workflow。
