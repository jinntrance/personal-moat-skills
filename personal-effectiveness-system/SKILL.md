# Personal Effectiveness System

目标：把零散的个人提效请求路由到互斥且穷尽的处理模块，避免“任务、时间、知识、阅读、复盘”混在一起。

## MECE 总框架

| 模块 | 解决的问题 | 输入 | 输出 | 排除边界 |
|---|---|---|---|---|
| 1. Capture 收集 | 信息、想法、任务散落 | 原始 inbox、聊天记录、会议记录、临时想法 | 去重后的 inbox 清单 | 不排序、不承诺执行 |
| 2. Clarify 澄清 | 不知道这件事到底是什么 | inbox 条目、背景、约束 | 项目 / 动作 / 等待 / 资料 / someday 分类 | 不做长期目标判断 |
| 3. Prioritize 取舍 | 事情太多，不知道先做什么 | 已澄清清单、目标、截止期、能量 | Now / Next / Later / Drop | 不排具体日程 |
| 4. Plan 计划 | 目标太大，不知道怎么推进 | 目标、项目、约束 | 里程碑、下一步、风险、验收标准 | 不替代执行现场反馈 |
| 5. Execute 执行 | 拖延、卡住、注意力分散 | 当前任务、卡点、可用时间 | 25/50/90 分钟执行块、阻塞解除动作 | 不重开战略讨论 |
| 6. Review 复盘 | 做完了但没沉淀 | 周期记录、结果、偏差 | 继续 / 停止 / 开始、系统调整 | 不责备用户 |
| 7. Knowledge 知识资产 | 内容很多但不可调用 | 笔记、经验、模板、输出 | 可检索笔记、checklist、prompt、SOP | 不收藏无触发条件的材料 |
| 8. Learning 阅读学习 | 读书、课程、资料内化 | 书名、章节、批注、读书笔记 | 分流、理解检验、资产化或结构化归档 | 深读走专用阅读工具 |

## 路由规则

1. 先问“用户当前卡在哪个环节”：收集、澄清、取舍、计划、执行、复盘、知识资产、阅读学习。
2. 如果请求跨多个环节，按上表从前到后串行处理；不要同时优化所有环节。
3. 每次输出必须包含：当前模块、判断理由、最小下一步、完成标准。
4. 如果用户给的是书名、阅读目标、章节困惑、读后消化请求：转用 `ai-era-reading`。
5. 如果用户给的是已有读书笔记并要求脑图/大纲/归档：转用 `reading-notes-organizer`。

## 模块工作流

### 1. Capture 收集

- 合并重复项。
- 每条只保留一个动词或一个信息点。
- 输出为 `inbox`，不做优先级承诺。

### 2. Clarify 澄清

把每条 inbox 分成五类：

- **项目**：需要多步完成，有明确结果。
- **下一动作**：15–60 分钟可完成的具体动作。
- **等待**：需要别人响应。
- **资料**：未来可能参考，但当前不行动。
- **Someday**：想法有价值，但近期不承诺。

### 3. Prioritize 取舍

使用四象限但避免空泛：

- **Now**：影响目标且有时效，今天或本周必须推进。
- **Next**：重要但可以排队。
- **Later**：有价值但当前窗口不适合。
- **Drop**：不符合目标、收益低或维护成本高。

每个 Now 项必须有下一动作和完成标准。

### 4. Plan 计划

对目标或项目输出：

```markdown
目标：
成功标准：
里程碑：
下一动作：
风险与预案：
本周只做：
```

### 5. Execute 执行

当用户拖延或卡住时，先定位卡点：

- 不清楚下一步 → 回到 Clarify。
- 太大 → 切成 25 分钟动作。
- 情绪阻力 → 降低启动门槛到 2 分钟。
- 外部阻塞 → 生成等待清单和替代动作。

输出一个单块执行计划，不要给长篇建议。

### 6. Review 复盘

复盘只回答四问：

1. 本周期真正完成了什么？
2. 哪些承诺不该接？
3. 哪个系统摩擦最大？
4. 下周期只调整一个什么？

### 7. Knowledge 知识资产

知识沉淀必须有触发条件：

- `当我遇到 ____ 时，调用 ____。`
- 产物优先级：checklist > prompt > SOP > 长笔记。
- 如果是读书笔记归档，转 `reading-notes-organizer`。

### 8. Learning 阅读学习

- 选书、读前、读中、读后理解检验：转 `ai-era-reading`。
- 已有读书笔记结构化：转 `reading-notes-organizer`。
- 非书籍学习材料（课程、论文、文章）按同样逻辑：先理解检验，再资产化，最后归档。


## 迭代空间判断

当用户问“还需要补什么”或“这个系统哪里不完整”时，按以下顺序评估，不要直接新增工具：

1. **是否缺示例**：如果已有规则但用户不知道怎么用，优先补端到端示例。
2. **是否缺模板**：如果产物格式反复出现，优先补 checklist / prompt / SOP 模板。
3. **是否缺校验**：如果输出容易漂移，优先补测试、脚本或固定验收标准。
4. **是否真有新边界**：只有当新需求无法归入 Capture / Clarify / Prioritize / Plan / Execute / Review / Knowledge / Learning 时，才建议新增 Skill。

输出补充建议时必须给优先级：P0 立即补、P1 下一轮补、P2 观察后补。

## 输出模板

```markdown
## 当前模块
[Capture / Clarify / Prioritize / Plan / Execute / Review / Knowledge / Learning]

## 判断理由
[为什么是这个模块]

## 处理结果
[清单、计划、决策或资产]

## 最小下一步
[一个可执行动作]

## 完成标准
[如何判断这一步完成]
```
